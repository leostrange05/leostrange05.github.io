    OOPre 2025 课程总结与回顾

    约 1883 个字 1 张图片 预计阅读时间 6 分钟

    转载来源

    本文转载自我的 CSDN OOPre 2025 课程总结与回顾

    一、架构设计¶

    1. 继承与接口设计¶

    • Equipment 和 Bottle 为 Item 类的子类，

    • Equipment 包含 Weapon 和 Armour，

    • Weapon 包含 Magicbook 和 Sword，

    • Bottle 包含四种类型的药水瓶，

    • Spell 包含 AttackSpell 和 HealSpell，

    • Bottle 和 Spell 继承了接口 Usage，可以调用 use 方法。

    2. 类的功能设置¶

    在最具体的可用物品类，例如 HealSpell、ManaBottle中，设置 use 方法，实现物品使用的条件检查反馈和使用执行。

    在 Adventurer 类中集成了单个冒险者自身的行为、冒险者对其他人的“一对一”操作、物品使用的前置条件判断、雇佣关系判断等操作。 例如，冒险者攻击另一个冒险者、冒险者增减体力值等。在这些方法中整合了金钱计算、死亡处理以及援助实现等功能。

    MainClass 类中的方法主要是涉及多个冒险者的统一处理，例如战斗过程计算联合防御值等。输入解析也在这个类中完成。

    Parser 在解析表达式的同时，调用已有的添加关系方法来导入关系。

    3. 设计模式的使用¶

    采用工厂模式 Factory 类来进行 Item 和 Spell 的对象创建，以集中管理类型多样的物品和法术。

    采用观察者模式实现雇佣关系以及援助。当需要援助时，递归地通知所有下级进行援助；当冒险者死亡时，通知上级和所有直接下属删除雇佣关系。

    4. 容器的使用¶

    容器 具体应用场景 原因
    HashMap 冒险者列表、冒险者拥有的物品、冒险者学习的法术 需要通过 id 来定位相应的对象
    LinkedHashMap 冒险者携带的药水瓶 在”id 索引“功能的同时，需要实现“容量上限”和“先进先出”功能，需保证存储顺序
    HashSet 冒险者的下属列表 可以方便地添加和移除
    ArrayList lr 命令的词法分析产生的 token 列表 简单的线性存储结构

    5. 迭代过程中的架构调整¶

    (1) 随着对容器的进一步学习和了解，以及迭代任务对于不同存储内容的功能要求，笔者从第三次作业全用 ArrayList 存储迭代为第五次作业中对不同的功能选用不同的容器。

    (2) 最初笔者将 Spell 也作为 Item 的子类，直接与携带的物品存储在一起，以方便 use 的实现。在之后的迭代任务中，这样的设计逐渐无法满足设计需求，且本身也不符合逻辑，因此最终删除了这个继承关系，并将 Spell 与 Item 分开存储。

    (3) 学习继承关系后，笔者将物品之间的关系用继承联系了起来，但是“创建物品”或者说实例化的过程仍然需要进行复杂的分类讨论。学习工厂模式后，笔者将这样的实例化过程集成在工厂类中，使得 Adventurer 类中的代码能够更加简洁清晰。

    二、使用 JUnit 的心得体会¶

    以往的程序测试往往是直接构造输入样例，通过输出结果定位错误。然而这样的方法既难以完整覆盖所有程序过程，也无法快速定位错误的位置，对于 Java 项目这种代码量巨大的程序显示出较大的弊端。

    JUnit 能够对每个类、每个方法单独检验正确性，在使用过程中，笔者感受到了其方便与精准，也不止一次地从 JUnit 测试中发现代码编写的错误。庞大的项目开发，必然不能保证所有的步骤万无一失。对于逻辑复杂、分支众多的程序，直接构造全局的测试很难覆盖所有的行、方法与分支。通过 JUnit 模块，我们能够直观地了解测试的覆盖率，保障测试的有效性。

    当然，JUnit 构造的测试也并不一定能够完全找出所有的 bug，但至少其能够为程序的正确性多一份保障。

    很遗憾，笔者在这几次的项目开发过程中，并未充分发挥 JUnit 单元的全部作用。很多时候为了快速完成任务，笔者往往只关注覆盖率尽量高而忽略了对于 Assert 语句的使用。在实际的开发过程中，不仅覆盖率需要保证，还需要与预期的效果进行比对。在以后的学习中，笔者会进一步完善 JUnit 的使用。

    类似软件或者游戏的开发，我们固然可以通过“内测”或者“公测”来检验 bug，但如果在开发阶段能及时发现 bug 并修复，将会省下大量的工作量，减少不必要的麻烦。

    三、学习OOPre的心得体会¶

    从面向过程到面向对象，笔者感受到了巨大的思想变化。

    • 以往的编程，“能用就行”，只要能够达成效果，而无需考虑可扩展性、安全性、易读性等诸多限制。这其实也是由于代码量不够大，而导致编程的开发十分随意。学习面向对象后，笔者在编程中开始考虑迭代开发的扩展性、成员和方法的安全性，开发的过程逐渐有了规矩。在团队的合作开发中，规范的开发对于协同工作是无比重要的。

    • 同时，面向对象这种模块化、层次化的思想，也让笔者对于庞大项目的构建有了认知。合理地设计对象层次结构，分派每个对象管辖的范围、负责的任务，都是使得项目结构清晰易懂、开发过程有迹可循的关键。

    笔者在这几次的开发中，对于结构的设计仍不成熟。或许是保留了“面向过程”的一些习惯，抑或是为了减少重构的工作量，Adventurer 类的功能过于集中。虽然类间的层次结构较为简单、控制流比较连续，但是类内的功能太多太乱，其实不甚符合面向对象的设计思路。目前的程序复杂度尚可以接受，但如果增加更多的功能呢？因此对于层次结构的设计，还需要继续打磨，既不能设计太多冗余的类，也不能让一个类内的方法过于复杂，这也是一种权衡的设计艺术。对此，希望笔者在 OO 正课能够继续完善。

    另外，前几次课对于 Git 的教学也让笔者体会到了版本控制工具的强大。尽管在课程的要求下，只需要提交和推送功能即可完成作业提交，但在多人协同、复杂迭代场景下，仍需要学会使用 Git 的进阶功能。希望在以后的应用过程中能够继续熟悉。

    四、对OOPre课程的简单建议¶

    希望在课程中的某些阶段，能够进行一个“设计结构评析”或者是给出参考的“优秀设计”，让同学能够及时调整自己的设计架构，减少冗余、优化结构，养成良好的“设计审美”。
